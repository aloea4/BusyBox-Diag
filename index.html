研究背景與動機

嵌入式系統診斷的三個根本困境

01

部署複雜度

top、df、ss 各自依賴獨立的動態函式庫。在容量受限的 rootfs 中，分別安裝的開銷往往超出可接受範圍，且版本相依管理複雜。

02

介面不一致

各工具的 CLI 介面與輸出格式差異顯著，難以整合至統一的自動化監控框架或 shell pipeline 流程中。

03

資源消耗過大

傳統互動式工具（如完整版 top）具有 ncurses 介面，在僅需擷取單次快照的場景下，啟動開銷與記憶體佔用明顯過大。

系統架構

嚴格四層分離，資料由核心向上單向流動

Kernel 介面層
系統真實狀態來源

直接讀取 Linux kernel 匯出的介面，不依賴任何私有 API 或外部函式庫。

/proc/net/tcp・udp /proc/[pid]/stat・status /proc/stat・meminfo・mounts statfs() syscall
libdiag 量測層
共用量測基礎設施（Zero-malloc 設計）

封裝 procfs 字串解析與 statfs 系統呼叫容錯，輸出強型別 C 結構體，讓三個工具不需各自重複實作底層解析邏輯。

困難與解決方案

開發過程中遭遇的三類設計難題

ptop

單次 procfs 讀取無法表示 CPU 使用率

/proc/[pid]/stat 的 utime、stime 是累積 jiffies，直接換算會讓閒置行程呈現虛高的使用率。

→ 採用 Snapshot-Delta 架構，擷取兩次快照的 jiffies 差值推導真實負載。以 bsearch() 依 PID 配對，穩定處理行程動態生滅。

diagfs

Docker 環境下掛載點視角落差

讀取 /proc/mounts 會取得宿主機掛載表，被容器特有的虛擬檔案系統污染，導致狀態誤判。

→ 改用 /proc/self/mounts，賦予程式 Mount Namespace Awareness，確保容器與實體機環境均能反映正確視角。

diagnet

/proc/net/tcp IP 位址位元組序錯誤

核心序列化時直接以十六進位印出 host-order 的 s_addr，在 x86 平台為 little-endian，導致 127.0.0.1 被誤讀為 1.0.0.127。

→ 實作 diag_ipv4_hex_le_to_str，解析為 unsigned int 後自 LSB 依序提取位元組還原，函式名稱保留 _le_ 後綴作為合約標記。

diagnet

--stats 模式記憶體無界成長

「全收集 → 排序 → 統計」流程在連線數龐大的主機上累積大量記錄，與嵌入式場景的記憶體預算衝突。

→ 串流導向處理（Stream-oriented），在 callback 階段直接累加計數，條件式跳過明細儲存，空間複雜度從 O(N) 退化至 O(1)。

ptop

互動式顯示與 shell pipeline 相容性衝突

ANSI 色彩與控制字元一旦被 pipe 傳遞至 awk，造成資料污染，破壞可解析性。

→ 以 isatty(STDOUT_FILENO) 自動偵測輸出環境，TTY 走 ptop_render_tty()，非 TTY 強制退回 batch 模式，兩種情境均兼顧。

diagnet

效能對照基準不對等導致比值失真

早期版本以 ss -tn 對照，但 ss 預設排除 LISTEN 與 TIME_WAIT，而 diagnet 列出所有狀態，工作量不對等，v2 比值達 1.95。

→ 自 v4 起改為 ss -tna / ss -tnua，並同步將 -O3 -flto 傳遞至 libdiag，修正後 v5 三組比值均降至 1.50 以內。
